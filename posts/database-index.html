什么是索引

索引是数据库中用于加速查询的数据结构，类似书的目录。没有索引，数据库需要全表扫描；有了索引，可以直接定位数据。

-- 创建索引
CREATE INDEX idx_user_email ON users(email);

-- 查看索引
SHOW INDEX FROM users;

为什么用 B+ 树

主流数据库（MySQL、PostgreSQL）使用 B+ 树作为索引结构：

  • 高度低： 3-4层即可支持千万级数据
  • 范围查询好： 叶子节点形成链表，适合范围扫描
  • 磁盘友好： 节点大小对齐磁盘块，减少IO

对比其他结构：

结构优点缺点适用场景
B+树范围查询优秀更新稍慢通用索引
哈希精确查询极快不支持范围等值查询
位图空间极小不适合高基数状态、性别等
全文文本搜索维护复杂文章内容

索引类型

1. 主键索引（聚集索引）

数据直接按主键顺序存储，一张表只能有一个。

CREATE TABLE users (
  id INT PRIMARY KEY,  -- 自动创建主键索引
  name VARCHAR(50)
);

2. 普通索引（非聚集索引）

存储索引列值和主键指针，通过指针回表查完整数据。

CREATE INDEX idx_name ON users(name);

3. 唯一索引

索引列值必须唯一，允许一个NULL。

CREATE UNIQUE INDEX idx_email ON users(email);

4. 联合索引

多个列组成的索引，遵循最左前缀原则。

CREATE INDEX idx_name_age ON users(name, age);

-- 能用上索引
SELECT * FROM users WHERE name = 'Tom';
SELECT * FROM users WHERE name = 'Tom' AND age = 20;

-- 不能用索引（跳过了name）
SELECT * FROM users WHERE age = 20;

5. 覆盖索引

索引已包含查询所需字段，无需回表。

-- 假设有索引 idx_name_age(name, age)
-- 这个查询可以直接从索引获取数据，无需回表
SELECT name, age FROM users WHERE name = 'Tom';

什么时候用索引

适合建索引的列：

  • WHERE、JOIN、ORDER BY 中频繁使用的列
  • 区分度高的列（如手机号 > 性别）
  • 数据量大但查询少的表不需要

不适合建索引的列：

  • 频繁更新的列（索引也要更新，增加写开销）
  • 数据量小的表（全表扫描更快）
  • 区分度低的列（如性别、状态只有两三个值）

索引失效场景

-- 1. 使用函数
SELECT * FROM users WHERE YEAR(created_at) = 2026;

-- 改为范围查询
SELECT * FROM users WHERE created_at BETWEEN '2026-01-01' AND '2026-12-31';


-- 2. 类型转换
SELECT * FROM users WHERE phone = 123456;  -- phone是字符串
-- 改为
SELECT * FROM users WHERE phone = '123456';


-- 3. LIKE 以 % 开头
SELECT * FROM users WHERE name LIKE '%Tom';
-- 改为
SELECT * FROM users WHERE name LIKE 'Tom%';


-- 4. OR 连接的字段没有都建索引
SELECT * FROM users WHERE name = 'Tom' OR age = 20;
-- 如果name有索引但age没有，索引失效


-- 5. NOT、!=、<> 等否定操作
SELECT * FROM users WHERE status != 1;

索引优化建议

  1. 优先在 WHERE 子句的列上建索引
  2. 联合索引遵循最左前缀，把区分度高的放前面
  3. 避免 SELECT *，只查询需要的列（利于覆盖索引）
  4. 区分度低的列不单独建索引，可以联合其他列
  5. 定期分析索引使用情况，删除无用索引
-- 分析SQL执行计划
EXPLAIN SELECT * FROM users WHERE name = 'Tom';

-- 查看索引使用情况（MySQL）
SHOW STATUS LIKE 'Handler_read%';
索引是把双刃剑：能大幅提升查询速度，但会增加写入开销和存储空间。要根据实际业务场景权衡，避免过度索引。
